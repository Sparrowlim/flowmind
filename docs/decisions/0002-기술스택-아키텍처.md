## 1. 배경 (왜 이 결정을 했나)
- 와이어프레임 완료 후 제품 구조 확정 단계(plan §7-3).
- 제약: 1인 · 6개월 · 저비용 · ADHD 개인용 · 모바일 세로.
- 전제 전환: **"제품 우선"**. 이력서용 헤지(인지도 높은 프레임워크, 불필요한 백엔드)를 배제.
  매일 쓰는 검증된 제품 + "시작 전환율" 케이스스터디가 최선의 채용 신호 → 제품 최적 = 채용 최적.
- "MVP에서 멈추지 않을 수 있다"는 전제로 **확장성**을 평가 기준에 포함. 판단 축 = *지금 잘 되는가*가 아니라 *나중에 문을 못 닫게 만드는가*(일방통행 문 vs 양방향 문).

## 2. 결정된 기술 스택 (Decided)
| ID | 층 | 선택 | 이유 | 버린 대안 |
|---|---|---|---|---|
| T1 | 언어 | **TypeScript** | 엔진 로직을 타입으로 안전하게, 테스트 용이 | JS |
| T2 | UI | **React** | 와이어프레임이 곧 컴포넌트, 생태계 | Vue/Svelte |
| T3 | 빌드 | **Vite** | 로컬 SPA엔 SSR 가치 0, 가볍고 빠름, Capacitor 궁합 | Next.js |
| T4 | 저장소 | **로컬 우선 — IndexedDB(Dexie)** | 즉시·오프라인·프라이버시, 결정마비 앱에 스피너 0 | Supabase(현 단계 과함) |
| T5 | 플랫폼 | **웹 + Capacitor 래핑** | 웹 코드 그대로 + thesis-critical 네이티브 2개만 브리지 | 순수 PWA / RN |
| T6 | 테스트 | **Vitest** | 엔진 단위 테스트 = 핵심 자산 | Jest |

- **한 줄 요약**: 웹(Vite+React+TS)으로 짜서 로컬(IndexedDB)에 저장하고, **하루 1회 체크인 알림**과 **백그라운드 지속 타이머** 두 가지만 Capacitor로 빌려온다.
- T5 근거: 두 기능은 thesis의 심장(하루 1회 체크인 · 시간맹)인데 순수 PWA(특히 iOS)에선 신뢰성 있게 동작 불가 → 도그푸딩 자체가 가짜가 됨.

## 3. 계층 구조
```
ui  →  core  ←  data
            platform → ui
```
의존 방향: `ui → core ← data`, `platform → ui`. **core는 아무것도 import 하지 않는다.**

| 디렉터리 | 역할 | 비고 |
|---|---|---|
| `core/` | 순수 로직: engine(캐스케이드·신호·타이브레이크), state(전이·쪼개기) | UI·IO·전역시간 의존 0 |
| `data/` | Repository 추상화: local(Dexie) / events(D6 append-only) | 로컬↔클라우드 교체점 |
| `ui/` | React 화면(와이어프레임) · components · hooks | **구현 시 `src/`로 명명**(Vite 기본 엔트리, 별도 `ui/` 디렉터리 안 만듦) |
| `platform/` | Capacitor 브리지: notifications · timer | 네이티브 2개만 |

- **핵심 설계 원칙(A1): 엔진을 순수 코어로 격리.** `pickNextCard(pool, now, cfg) → Decision`을 React·저장소·전역시간에서 분리.
  - 효과: ⒜ 골든 테스트로 D13(결정 재현성) 증명 = 면접 자산, ⒝ v2 학습 엔진(D6 로그 연료)의 교체점, ⒞ 플랫폼·저장소 교체 시 코어 무손상.

## 4. 데이터 모델 (필드 수준 초안)
> 코드는 구현 단계에서. 여기서는 엔티티·필드와 근거만.

- **Task**: id(UUID) · title(유일 필수) · 부스터(deadline? · important · estimateMin?) · 바닥(createdAt · lastServedAt? · deferCount · splitRefuseCount) · 상태(state · dormantUntil? · parentId? · focusPausedAt? · accumulatedSec? · completedAt?) · 확장보험(updatedAt · deletedAt?)
- **DecisionEvent** (D6, append-only): taskId · servedAt · weekday · hour · ruleFired · ageSnapshot · deferSnapshot · action(start|done|snooze|skip|split) · elapsedSec? · completed?
- **Settings**: checkinTime · accentColor · handdrawn
- DecisionEvent가 §5 "시작 전환율" 지표 + v2 학습 엔진 연료를 동시 공급.

## 5. 확장성 — 일방통행 문 & 보험 (Decided)
미래 기능(plan v2 후보 · "안 만들 것" · ADR-0001 풀엔진) 천장 점검 결과: **플랫폼이 막는 칸 없음.** 위험은 *다기기 동기화 소급 적용* 하나로 수렴.

- E1. **동기화 친화 데이터 모델**(가장 중요, 지금 까는 싼 보험 — 기능은 미구현, 문만 개방):
  - id = **UUID**(자동증가 정수 금지) · 모든 엔티티 `updatedAt` · `deletedAt?`(soft delete) · `EventLog` append-only(= 미래 sync 로그) · `schemaVersion` 필드 · **"단일 기기 가정" 코드 금지**.
- E2. **순수 코어 격리**(§3 A1) → 플랫폼/엔진 교체 시 코어 이식.
- E3. **Repository 추상화**(T4) → 동기화는 두 번째 구현체 삽입.
- E4. **Capacitor 탈출 조기경보**: 위젯/워치/온디바이스 ML이 *핵심 루프*가 되거나, 플러그인 없는 네이티브가 동시 2개↑ 필요 시 RN 전환 검토. 그때도 core+UI 상당 이식 → "재작성"이 아닌 "껍데기 교체".

## 6. 빌드 순서 (Decided)
최적화 함수 = **"내가 매일 쓰기까지 최단 경로"**(이 앱은 본인 고통 해결 → 본인 실사용이 설계 나침반). §5 검증이 백로그를 쓴다(YAGNI, plan §1 "안 만들 것"은 성역).
1. **엔진 먼저(헤드리스)** — `core/engine` 순수 TS + 골든 테스트. 심장부터.
2. **내 폰 최소 루프** — 담기 → 지금 카드 → 시작/이따다시 → 집중 타이머 → 로컬 저장 + Capacitor 껍데기 + 하루 1회 알림. ← **본인 매일 사용 시작.**
3. **ADHD 깊이** — 쪼개기/자동쪼개기 + 미룸·재우기 상태머신 + 빈 상태 + 보관함. (1~2단계 도그푸딩이 무엇을 만들지 정함)
4. **측정** — D6 로깅 연결 → 시작 전환율로 §5② 2주 검증, 상수 튜닝.
5. **소수 테스트(3~5명) → 케이스스터디.**

## 7. 검증 단계 도구 (Decided)
지표(행동)와 설문(느낌)은 다른 데이터 → 다른 도구. **시작 전환율은 설문으로 못 받음 → 앱 이벤트 로그(D6)가 측정.**
- **1단계 — 본인 도그푸딩 2주**: 수집 도구 0개. 로컬 이벤트 로그 + **"내 데이터 내보내기"(JSON/CSV)** 하나만. 백엔드 없음.
- **2단계 — 3~5명 테스트**: **PostHog**(서빙→시작 퍼널 자동) + **구글폼**(정성 5~7문항). 둘 다 그때 추가.
- 미뤄도 안전한 이유: 로그는 1단계부터 로컬에 쌓고 추상화 뒤에 있으므로, PostHog는 *두 번째 출구를 꽂는 일*(재작업 0). 로컬 우선(T4) 불변.
- **프라이버시**: 폰 밖 전송 이벤트엔 할 일 제목 등 내용 제외(규칙·행동·시각만), 테스터 동의. = 프라이버시 우선 서사.

## 8. 열린 결정 (Open)
- T-O1. △ 부분 CLOSED — **게이트 순서 = 이어하기 우선(B)** 확정(engine-spec §6 S1: GATE-RESUME→GREETING→SPLIT). 잔여: "집중 일시정지 중 새 캡처" 시나리오의 사용자 선호만 PENDING(골든 G-30), 도그푸딩(§5②)에서 잠금. 구현은 잠정값 B로 진행 가능.
- T-O2. **iOS 빌드 경로 — 알림 게이트까지 결정 보류(양방향 문).** 개발 환경이 Windows라 Capacitor iOS 네이티브 빌드(서명·기기설치)에 macOS/Xcode가 필요 → 본인 아이폰 도그푸딩(§6 빌드순서 2단계)의 새 마찰점.
  - **프레이밍**: iOS의 어려움은 전부 *도구·비용* 문제지 *제품·능력* 문제가 아님(Capacitor iOS는 thesis 요구 기능—하루1회 알림·타이머—을 모두 지원). → 제품 설계를 흔들 이유 없음, 지금 풀 필요도 없음.
  - **단계별 가용성**: 1단계 엔진(헤드리스) = iOS 무관, 즉시 착수(Mac·$99·기기 전부 불필요). 2단계 UI의 ~90% = 아이폰 Safari PWA로 무료 도그푸딩 가능(경과 타이머는 `startedAt` 타임스탬프 계산식으로 백그라운드 실행 없이 동작). **Mac/$99 뒤에 갇힌 건 "하루 1회 체크인 알림" 하나** = 검증 순서상 *마지막* 조각.
  - **결정 규칙**: 이 비용은 알림 게이트(2단계 후반)에 닿을 때, *몇 주치 실사용 증거를 들고* 판단. 제품이 스스로를 증명한 뒤에만 지출 → 문제 아니라 필터.
  - **선택지(그때)**: ⒜ 현 스택 유지 + 클라우드 Mac/CI 빌드(예: GitHub Actions macOS 러너 / Codemagic) + Apple Developer **$99/년**. ⒝ 출시 단계엔 중고 Mac mini 상시 보유 검토. — "폰에 오래 설치/배포"엔 $99/년은 어느 경로든 사실상 불가피.
  - **하지 말 것**: Expo/RN 전환. *싼 도구 문제(Mac 없음)*를 *비싼 UI 재작성(웹 React → React Native)*과 맞바꾸는 손해. 와이어프레임·엔진스펙·ADR이 전부 웹 기준 확정 + Mac 문제는 Capacitor도 클라우드 빌드로 해결됨. (E4 RN 탈출 조기경보와 별개 — 현 단계 전환 근거 없음.)
  - **상시 리스크(출시 시)**: Windows에서 iOS 빌드를 계속 굽는 의존(클라우드 Mac 또는 Mac mini)이 운영 비용으로 잔존. MVP 문제 아님, 출시 단계 문제.
- 엔진 구현 스펙(타입·절차·골든 케이스)은 **docs/specs/engine-spec.md (FINAL v1.0)** 으로 분리 확정 — 빌드 1단계 착수 준비 완료.

## 9. 결과/영향 (Consequences)
- (+) 플랫폼 천장이 PWA-only보다 높고, 탈출도 저비용(E4).
- (+) 가장 위험한 엔진을 스택 결정과 무관하게 즉시 착수 가능(순수 TS).
- (+) 제품 우선 선택이 채용 서사(시작 전환율 케이스스터디)와 정렬.
- (−) 다기기 동기화는 데이터 모델 보험(E1) 준수가 전제 — 안 지키면 소급 적용이 비쌈.
- (−) Capacitor는 일부 깊은 네이티브 기능에서 플러그인 의존 → E4 조기경보로 관리.
