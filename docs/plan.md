# Part 1 · 제품 개요 (왜)

## 1.1 배경 & 고통

- **목표**: 6개월 내, 신입 웹 개발자 포트폴리오용 + 강한 채용 스토리 제품을 1인·저비용 완성.
- **개인 내러티브(핵심 자산)**: 가족이 ADHD 진단, 본인도 유사 어려움 → "우리 같은 사람을 위해 만들었다"(의료적 주장 X).
- **제작자가 실제 겪는 고통(근거)**:
  1. 할 일 많으면 머리 멈춤·우선순위 못 정함(결정 마비)
  2. 미리 안 정하면 아무것도 안 함, 있어도 마감 직전까지 미룸
  3. 적어둬도 리스트를 다시 안 봄
  4. 푸쉬 폭탄 → 전부 무시 → 중요한 것도 놓침
  5. 몰입하면 시간 순삭(시간맹)

## 1.2 핵심 통찰

- 저장 공간이 아니라 **이미 적어두고 무시함** → "지금 이거 하나"를 대신 정해주고 시작시켜야 한다.
- **리프레이밍**: 경쟁자는 정교한 알고리즘이 아니라 "멈춰버린 뇌". 엔진은 완벽할 필요 없이 "합리적 하나"가 "못 정함"을 이기면 됨 → 규칙 캐스케이드면 충분, 6개월 내 완성 가능.

## 1.3 검증 전략 (riskiest-assumption-first)

- **가장 위험한 가정**: "사용자가 위임을 신뢰하고 카드가 시킨 일을 실제로 시작하는가?"

| 단계                | 내용                                             | 측정             |
| ------------------- | ------------------------------------------------ | ---------------- |
| ① 얇은 슬라이스     | 캡처+엔진+지금 카드+왜 지금?+지금 시작/이따 다시 | —                |
| ② 본인 도그푸딩 2주 | 매일 실사용                                      | **시작 전환율**  |
| ③ 커뮤니티 3~5명    | 소수 테스트                                      | 신뢰·스누즈 패턴 |

- 얇은 슬라이스 = MVP 골격(일석이조). "시작 전환율로 가설 검증" = 강력한 면접 자산.
- 추가 검증 포인트: 보관함(풀)을 자주 들여다보며 다시 압도되는가(설계원칙1 긴장 확인).

## 1.4 성공 기준

- [ ] 제품 정의가 즉시 이해됨
- [ ] 엔진이 설명 가능한 "다음 하나"를 안정적으로 산출
- [ ] 리스트·푸쉬 없이 시작하게 만듦(시작 전환율 검증)
- [ ] 6개월 내 출시 + 실사용자 + 케이스스터디 작성

# Part 2 · 제품 요구사항 (무엇을)

## 2.1 제품 정의

| 항목     | 내용                                                   |
| -------- | ------------------------------------------------------ |
| 한 명    | 우선순위에서 멈추고 리스트·푸쉬를 무시하는 사람(=본인) |
| 한 순간  | 할 일이 쌓여 얼어붙은 순간                             |
| 한 Job   | "지금 이거 하나만 해"라고 정해주고 시작시켜줘          |
| 한 화면  | 카드 한 장 → [지금 시작]/[이따 다시]/[쪼갤래요]        |
| MVP 심장 | 우선순위 자동 결정 + 카드 한 장                        |

### 설계 원칙 (thesis)

1. **리스트를 안 보여준다**(목록=압도) — 보관함(풀)은 '가벼운 열람'으로만 분리 노출
2. **우선순위는 앱이 정한다**(결정 마비 회피)
3. **푸쉬 폭탄 금지** → 하루 1회 의도된 체크인(설정에서 하루 1회 고정, 켜도 알림 안 늘어남)
4. **시작하면 경과시간 크게**(시간맹)
5. **죄책감 0** — 빨간 배지/압박 금지

### 스코프

- **만들 것(MVP)**: 빠른 캡처 / 캐스케이드 엔진 / 카드 한 장 / 시작 시 경과시간 / 하루 1회 체크인 / 쪼개기
- **안 만들 것(MVP — 성역)**: 협업, 프로젝트 트리, 태그, 캘린더 동기화, 통계 대시보드, 복잡 반복, 외부 메신저 연동, "다른 거" 버튼, 가용시간 모드
- **v2 후보**: 가용시간 "N분 있어" 필터, 점수식 정교화, AI 쪼개기

## 2.2 UX 흐름

### 지금 카드 (유일한 결정 지점)

- 구성: 제목 / (입력된)예상시간·마감 / "왜 지금?" / **[지금 시작][쪼갤래요][이따 다시]** — 선택지 정확히 3개 (죄책감 0 톤)
- **점진적 공개**: 기본은 접힌 카드(제목+"왜 지금?" 한 줄+지금 시작). '자세히'로 펼치면 캐스케이드 근거·예상시간·마감·중요도 노출
- **지금 시작** → 집중 화면(경과시간 크게), 완료/일시정지
- **이따 다시** → 다음 후보 서빙(='거부+다음' 겸함, 별도 버튼 불필요). 스누즈 시간은 앱이 정함, 미룬 횟수 +1. 카드에 "N번째 미룸 · 다음 체크인 시각" 단서 표기
- **쪼갤래요** → 쪼개기(첫 1스텝만)
- **왜 지금?** → 선정 이유 한 줄(신뢰 확보 핵심, §2.3 카피 세트)

### 보조 흐름

- **캡처**: + 한 번 → 제목만 필수 → (선택)마감·중요·예상시간(메타데이터 피커) → 끝. 미입력은 카드에서 나중 보충
- **완료 후**: 짧은 축하 → [한 개 더 할래요](주)/[오늘 그만](보조). 자동 연속 서빙 X
- **일시정지 → 재개**: 타이머만 멈춤, 누적 경과시간을 0으로 리셋하지 않고 이어 받음
- **하루 1회 체크인 알림**(아침)

### 빈 상태 (죄책감 0)

- 0개: "다 비웠어! 🎉"
- 전부 스누즈: "지금은 쉬어도 돼. 다음 체크인 때 다시 보여줄게."

## 2.3 우선순위 엔진 — 규칙 (제품 관점)

> **방식**: 규칙 캐스케이드. 점수식(설명 불가·튜닝 지옥) 불채택, 컨텍스트 필터는 v2.
> **알고리즘 세부(상수표·pickNextCard 절차·타이브레이크·불변식)는 Part 3 → engine-spec.md / engine-flow.md 가 원본.** 여기선 제품이 약속하는 규칙·카피만.

### 입력 신호 모델 (2계층 — ADR-0001)

바닥은 항상 존재해 엔진이 "빈손"이 되지 않음(기아 방지), 부스터는 있으면 끌어올림.

- **바닥 신호(시스템·항상)** — 제목만 있어도 동작
  - **나이(age)**: 방치 기간. 단조 증가로 기아 방지(aging boost).
  - **미룸 횟수**: 이따 다시/넘어갈래 누적. 임계값(기본 3) 초과 시 규칙②(자동 쪼개기).
- **부스터 신호(사용자·선택 — 없으면 중립)**
  - **마감**: 임박할수록 가중 ↑(시간 감쇠).
  - **예상시간**: ≤2분이면 빠른 승리 도파민.
  - **중요**: **동률 타이브레이커**(주축 아님 — 점수 가산 아님). 같은 티어·같은 나이일 때 우선.
- **불변식**: ⒜ 바닥은 항상 존재 → 엔진은 빈손이 안 됨 ⒝ 부스터는 끌어올릴 뿐 바닥을 끄지 못함.
- **결정 이벤트 로깅**: MVP부터 의무(ADR-0001 D6) — 미래 학습 엔진의 데이터 연료.

### 캐스케이드 규칙 + "왜 지금?" 카피 (위에서부터 첫 매칭)

1. **펑크 방지** — 오늘 마감 + 중요
   - 접힌: `왜 지금? · 오늘 마감` / 펼친: `규칙 ① · 오늘 마감 + 중요 → 1순위로 골랐어요`
2. **회피 누적** — N회(기본 3) 이상 미룸 → 쪼개기 제안 (전용 화면 직행, §2.4)
   - 접힌: `왜 지금? · 세 번째 만남` / 펼친: `규칙 ② · 세 번째예요. 작게 쪼개서 시작해볼까요?`
3. **마감 임박 순**
   - 접힌: `왜 지금? · 내일까지` / 펼친: `규칙 ③ · 마감이 가까워요(내일) → 지금이 적기`
   - _aging boost 승격 카피_: 접힌 `왜 지금? · 너무 오래 묵었어요` / 펼친 `규칙 ③(승격) · 2주 넘게 잠자던 일 → 더 미루지 않게 지금 한 번`
4. **빠른 승리** — 2분이면 끝나는 일(도파민)
   - 접힌: `왜 지금? · 2분이면 끝` / 펼친: `규칙 ④ · 2분이면 끝나는 일 → 가볍게 하나 비우기`
5. **오래 방치 순**
   - 접힌: `왜 지금? · 12일째 방치` / 펼친: `규칙 ⑤ · 12일 동안 잠자던 일 → 한 번 꺼냈어요`
6. **그 외 중요도 높은 순**
   - 접힌: `왜 지금? · 중요 표시한 일` / 펼친: `규칙 ⑥ · 급한 건 없어요. 중요 표시한 일부터`

> 카피 원칙: 같은 카드, 사유만 6가지 → **투명성이 위임 신뢰의 핵심**. (초기엔 단순하게, 도그푸딩 후 순서/임계값 정교화.)

## 2.4 ADHD 특화

### 미룸의 역설 → 앱이 먼저 쪼개기 제안

- 미룰수록 ↑(규칙②), 수치심 금지. 3회 초과 시 **앱이 먼저** 쪼개기 제안(사용자 호출 아님).
- **규칙② 전용 화면**: 미룸 > 임계값(기본 3)이면 일반 "지금 카드"를 **건너뛰고** 자동 쪼개기 제안 화면이 대신 노출(한 트리거 = 한 화면).
  - 화면: 파란 배지 `규칙 ② · 세 번째 만난 일이에요` + "미루는 데는 이유가 있어요. 더 작게 만들어 볼까요?" + 작은 첫 걸음 예시
  - 선택지 3개(주→보조→약한 거부): ① `좋아, 쪼개기` ② `아니, 그냥 시작할래` ③ `지금은 넘어갈래`
  - **거부 출구 유지**: 가장 많이 미룬 일이라도 '싼 거부' 출구를 막지 않음. '넘어갈래'도 미룸 +1 → 다음엔 더 작게 제안.
  - 카피 원칙: 수치심 금지 — '벌써/또' 등 책망 어휘 배제, 사실 진술("세 번째예요")로.

### 미룸 횟수 정책

- **증가**: 이따 다시/넘어갈래 → `defer_count += 1`, 동시에 나이 리셋(나이·미룸 이중 발동 방지)
- **리셋(→0)**: 시작(집중 진입) 또는 쪼개기 수락(= 의미 있는 관여 = 회피 종료)
- **루프 차단**: 자동 쪼개기에서 `넘어갈래` → 쿨다운 재우기. **2회 거부 → 장기 재우기(7일)**, 풀 복귀 시 리셋 → 무한 쪼개기 종결
- **시간 감쇠**: MVP 비포함(v2 튜닝 후보)

### 신뢰 = 정확도 아님

## 3.2 스택 한 줄 요약 (상세 → ADR-0002)

웹(**Vite + React + TypeScript**)으로 짜서 로컬(**IndexedDB / Dexie**)에 저장하고, **하루 1회 체크인 알림**과 **백그라운드 지속 타이머** 두 가지만 **Capacitor**로 빌려온다. 테스트는 **Vitest**.

## 3.3 계층 구조 한눈에 (상세 → ADR-0002 §3)

```
ui  →  core  ←  data
            platform → ui
```

- `core/` 순수 로직(engine·state) — **아무것도 import 안 함**(전역시간·IO·랜덤 0). 엔진은 골든 테스트로 결정 재현성 증명.
- `data/` Repository 추상화(Dexie / append-only 이벤트 로그) — 로컬↔클라우드 교체점.
- `src/` React 화면(와이어프레임).
- `platform/` Capacitor 브리지(알림·타이머) — 네이티브 2개만.
